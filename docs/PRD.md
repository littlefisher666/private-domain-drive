# 私人共享文件 App 需求文档

## 1. 文档概述

- 项目名称：基于阿里云 OSS 的轻量私人共享文件 App
- 项目名称：私域网盘
- 英文名称：Private Domain Drive
- 文档版本：V1.0
- 文档日期：2026-06-01
- 目标平台：Android、macOS
- 文档目标：定义一款面向 5 人以内小范围使用的轻量共享文件 App 的第一期需求、技术边界与实施约束，用于产品评估、技术选型与开发排期。

## 2. 背景与问题定义

### 2.1 当前背景

用户希望拥有一个适合小范围熟人使用的共享文件工具，核心诉求是：

- 文件统一存放在阿里云 OSS
- 成员数量少，预计不超过 5 人
- 总存储规模约 50GB
- 需要 Android 和 macOS 双端可用
- 不希望搭建传统重后端
- 不希望维护复杂的用户体系、数据库和细粒度权限模型

现有商业网盘虽然具备成熟能力，但存在以下问题：

- 指定成员共享文件夹的产品形态不完全符合预期
- 会员费用和功能打包方式不一定匹配小规模私人使用场景
- 数据和权限规则不可完全自定义
- 某些产品偏链接分享，不适合长期固定成员共享

### 2.2 核心问题

如何以较低成本构建一个轻量、可控、可跨端使用的私人共享文件 App，在不引入重型后端和复杂基础设施的前提下，满足小范围文件共享、上传、下载和预览需求。

### 2.3 目标用户

- 主用户：具备基本技术理解、愿意自行管理云资源的个人开发者或小型家庭/熟人群体组织者
- 次用户：被邀请加入共享空间的家人、朋友或固定协作者

## 3. 产品目标

### 3.1 目标

第一期产品需要满足以下目标：

- 支持 Android 和 macOS 使用同一套服务能力访问共享文件空间
- 支持固定少量成员访问同一个共享空间
- 支持文件浏览、上传、下载和基础预览
- 尽量不依赖传统 ECS、自建常驻服务和数据库
- 控制整体云资源成本在较低范围内
- 保持权限模型简单，减少维护成本

### 3.2 非目标

第一期不追求以下能力：

- 替代成熟商业网盘
- 面向公开用户开放注册
- 支持大规模用户、多租户、复杂协作
- 提供精细到文件级或成员级的复杂权限编排
- 提供视频在线播放转码、Office 在线编辑、秒传、内容审核等重能力
- 构建完整后台管理系统

## 4. 方案定位

本产品定位为：

`一个基于阿里云 OSS 的轻量私人共享文件 App`

不是：

`一个完整商业网盘`

该定位决定了产品设计应优先满足：

- 小规模可用
- 成本可控
- 架构简单
- 维护门槛低

而不是追求大而全功能。

## 5. 需求范围

### 5.1 In Scope

第一期范围包含：

- 成员登录与身份识别
- 获取临时访问凭证
- 浏览共享文件空间
- 创建和查看文件夹结构
- 上传文件
- 下载文件
- 预览部分常见文件
- 基于权限显示可用操作
- 跨端基础一致体验

### 5.2 Out of Scope

第一期明确不做：

- 公开分享链接管理
- 复杂邀请流程
- 站内消息通知
- 文件评论、标签、收藏
- 回收站
- 文件版本历史
- 审计日志中心
- 在线协同编辑
- 视频在线播放转码
- 复杂搜索
- 数据库驱动的成员/角色后台

## 6. 用户场景

### 6.1 家庭共享

家庭成员共同访问照片、证件扫描件、家庭文档、影音资料等内容。

### 6.2 熟人资料共享

固定的 2 到 5 人小群体共享学习资料、项目资料或常用文件。

### 6.3 多设备访问

用户在 Android 手机上查看、上传照片或文档，在 macOS 上整理目录、批量下载和拖拽上传文件。

## 7. 用户故事

### 7.1 核心用户故事

1. 作为一个共享空间成员，我希望登录后直接看到我有权限访问的文件列表，以便快速获取资料。
2. 作为一个共享空间成员，我希望在手机端上传照片或文件到指定目录，以便与其他成员共享。
3. 作为一个共享空间成员，我希望在 macOS 端拖拽上传文件，以便更高效地整理资料。
4. 作为一个共享空间成员，我希望在权限允许时看到上传、下载、删除等按钮，在权限不允许时不显示或禁用相关操作。
5. 作为一个共享空间管理员，我希望通过阿里云侧配置成员权限，而不是在 App 内维护复杂后台。

## 8. 产品方案概述

### 8.1 总体思路

采用轻后端方案：

- 文件存储：阿里云 OSS
- 身份与授权：阿里云 RAM / STS
- 轻量后端：阿里云函数计算 FC
- 客户端：Flutter 开发 Android 与 macOS 双端

### 8.2 为什么不采用纯客户端长期密钥方案

纯客户端直接嵌入固定密钥虽然开发更快，但存在明显风险：

- 密钥泄露风险高
- 无法优雅撤权
- 不适合多人使用
- 客户端本地 UI 控制不能代替云端权限控制

因此第一期建议保留一个极轻量的后端能力，仅用于签发临时凭证和返回少量配置，不引入传统重后端。

## 9. 功能需求

### 9.1 登录与身份识别

#### 功能描述

用户通过预设的身份体系完成登录，App 获取当前身份并换取访问 OSS 的临时凭证。

#### 需求说明

- App 支持用户登录
- 登录成功后调用轻后端接口获取 STS 临时访问凭证
- App 能识别当前用户身份和基础能力信息
- 凭证过期后支持自动刷新或重新获取

#### 验收标准

- 用户可在 Android 和 macOS 成功登录
- 登录后 3 秒内完成凭证获取并进入文件列表页
- 凭证失效后，App 能提示并恢复会话，不出现静默失败

### 9.2 文件浏览

#### 功能描述

用户可以查看共享空间内的目录和文件。

#### 需求说明

- 支持按目录层级浏览文件
- 支持显示文件名、大小、更新时间、文件类型
- 支持下拉刷新或手动刷新
- 空目录和空状态有清晰提示

#### 验收标准

- 目录切换响应顺畅
- 50GB 规模、正常目录结构下可稳定列出文件
- 文件列表与 OSS 实际内容保持一致

### 9.3 上传文件

#### 功能描述

有上传权限的用户可将本地文件上传到当前目录。

#### 需求说明

- Android 支持系统文件选择器上传
- macOS 支持文件选择与拖拽上传
- 显示上传进度、状态、失败重试
- 大文件需支持分片上传

#### 验收标准

- 小文件上传成功率高
- 网络波动时上传失败可重试
- macOS 可通过拖拽上传至少一个文件

### 9.4 下载文件

#### 功能描述

用户可将文件下载到本地设备。

#### 需求说明

- 支持选择下载保存位置或使用默认目录
- 显示下载进度和状态
- 下载失败支持重试

#### 验收标准

- Android 和 macOS 均可完成下载
- 下载成功后用户可在本地找到文件

### 9.5 文件预览

#### 功能描述

用户可预览部分常见文件。

#### 第一期开启范围

- 图片预览
- PDF 预览
- 文本预览

#### 第一期开启限制

- 不做 Office 在线编辑
- 不做复杂视频在线播放转码

#### 验收标准

- 图片可直接打开查看
- PDF 能正常加载
- 文本文件可显示基础内容

### 9.6 权限驱动的功能显示

#### 功能描述

App 根据用户当前能力渲染上传、下载、删除等入口。

#### 设计原则

- UI 层显示由 App 控制
- 最终访问控制以 OSS 云端鉴权结果为准
- 不在客户端单纯解析复杂 RAM Policy 作为唯一依据

#### 建议实现

- App 获取当前用户能力信息
- 对关键操作进行能力探测或服务端下发能力描述
- 根据能力结果渲染按钮

#### 验收标准

- 无权限操作不显示或不可点击
- 即使前端误显示，OSS 也能拒绝未授权请求

### 9.7 基础错误处理

#### 需求说明

- 网络异常提示清晰
- 凭证失效可感知
- OSS 权限不足时给出明确提示
- 上传、下载失败允许重试

## 10. 非功能需求

### 10.1 性能

- 登录后文件列表首屏加载尽量控制在 3 秒内
- 常规小文件上传下载响应稳定
- 文件列表滚动流畅

### 10.2 可用性

- 界面尽量简单，不暴露阿里云底层复杂概念给普通成员
- Android 和 macOS 的主要信息结构保持一致
- 操作反馈明确，包括加载、成功、失败、重试

### 10.3 安全性

- 不在客户端持久存储高权限永久密钥
- 使用临时凭证访问 OSS
- 临时凭证有效期应合理控制
- 删除操作需二次确认

### 10.4 成本约束

- 云资源方案需优先控制固定成本
- 不引入传统 ECS 作为第一期基础设施
- 尽量避免数据库作为第一期依赖

## 11. 技术需求与约束

### 11.1 技术栈建议

- 客户端：Flutter
- 文件存储：阿里云 OSS
- 临时授权：阿里云 STS
- 轻后端：阿里云函数计算 FC
- 可选接口入口：阿里云 API 网关

### 11.2 为什么选择 Flutter

- 支持 Android、macOS，后续也可扩展 Windows
- 一套项目可复用大部分业务逻辑与 UI 结构
- 适合做响应式布局，兼容手机和桌面不同尺寸

### 11.3 为什么暂不引入数据库

第一期没有强烈数据库需求，原因包括：

- 不需要复杂成员关系管理
- 不需要文件扩展元数据
- 不需要分享记录、回收站、审计日志
- 主要数据本体都在 OSS

后续若引入成员管理、备注、标签、审计等功能，再评估数据库。

### 11.4 为什么不采用传统 ECS 后端

- 第一期开销和运维复杂度偏高
- 与项目体量不匹配
- 轻后端足以承担 STS 凭证签发和轻量配置下发

## 12. 信息架构与数据组织建议

### 12.1 OSS 目录组织

建议采用按前缀组织的目录结构，例如：

```text
shared/
  common/
  photos/
  docs/
  uploads/
```

后续如需扩展用户个人空间，可增加：

```text
users/
  user-a/
  user-b/
```

### 12.2 状态持久化建议

- 文件数据：存于 OSS
- 函数临时运行数据：不作为持久化依赖
- 极少量配置：可先采用函数环境变量或 OSS 中的 JSON 配置文件

## 13. 平台适配要求

### 13.1 Android

- 支持系统文件选择器
- 适配移动端窄屏布局
- 处理前后台切换和网络变化
- 优先保证上传、下载、预览体验

### 13.2 macOS

- 支持宽屏布局
- 支持拖拽上传
- 支持更适合桌面的双栏或三栏布局

### 13.3 统一性要求

- 两端尽量共用业务逻辑
- UI 可根据屏幕宽度和平台做差异适配
- 不要求像素级完全一致，但信息结构和核心流程应保持统一

## 14. 权限模型建议

### 14.1 第一期开启的角色

- 管理员
- 普通成员

### 14.2 第一期开启的能力

- 浏览文件
- 下载文件
- 上传文件
- 删除文件

### 14.3 建议原则

- 权限尽量在阿里云侧约束
- App 负责能力呈现和交互收敛
- 复杂细粒度权限不纳入第一期

## 15. 关键流程

### 15.1 登录与进入文件空间

1. 用户打开 App
2. 用户完成登录
3. App 调用轻后端接口
4. 轻后端校验身份并返回 STS 临时凭证
5. App 使用临时凭证访问 OSS
6. App 拉取文件列表并渲染首页

### 15.2 上传流程

1. 用户选择文件
2. App 校验当前目录和能力
3. App 使用临时凭证发起上传
4. 上传过程中显示进度
5. 上传成功后刷新文件列表

### 15.3 下载流程

1. 用户点击下载
2. App 获取下载地址或直接发起下载请求
3. 下载完成后保存至本地
4. App 给出成功反馈

## 16. 成功指标

第一期可采用如下指标评估：

- 5 人以内成员均可稳定登录并访问共享文件空间
- 主要功能可用率达到预期，上传和下载能稳定完成
- Android 与 macOS 双端均可完成核心流程
- 总体云资源成本保持在低成本区间
- 无需部署传统 ECS 和数据库即可完成第一期运行

## 17. 风险与应对

### 17.1 权限设计与安全风险

风险：

- 客户端错误理解权限
- 凭证泄露导致越权访问

应对：

- 真实权限始终由 OSS/RAM 云端鉴权
- 使用 STS 临时凭证而非长期高权限密钥
- 控制临时凭证有效期

### 17.2 跨平台文件系统差异

风险：

- Android 和 macOS 文件访问方式差异较大

应对：

- 优先使用成熟跨平台插件
- 在设计阶段提前验证文件选择、保存、拖拽能力

### 17.3 传输稳定性风险

风险：

- 大文件上传下载在弱网环境下失败

应对：

- 使用分片上传
- 提供重试机制
- 做基础断点恢复能力设计

### 17.4 需求膨胀风险

风险：

- 项目逐步演化成完整网盘，超出原定范围

应对：

- 严格控制第一期范围
- 先完成轻量共享能力，再评估扩展功能

## 18. 分期建议

### 18.1 第一阶段：最小可用版本

- 登录
- STS 临时凭证获取
- 文件列表
- 上传
- 下载
- 图片/PDF/文本预览
- Android 与 macOS 基础适配

### 18.2 第二阶段：体验增强

- 上传下载任务管理
- 更好的缓存策略
- 更清晰的目录管理
- 更完善的错误提示

### 18.3 第三阶段：按需扩展

- 成员管理增强
- 分享链接
- 文件备注/标签
- 操作日志
- Windows 端支持

## 19. 开发前确认项

在正式开发前，建议确认以下事项：

- 登录方式的最终选型
- 阿里云侧 RAM / STS 授权模型
- 是否需要 API 网关
- Android 与 macOS 所选 Flutter 插件兼容性
- 上传、下载和预览的最小文件类型范围
- 删除能力是否第一期开放给所有成员

## 20. 结论

本项目适合采用“Flutter + OSS + STS + FC”的轻量方案，以较低成本实现一个面向 5 人以内、50GB 量级的小范围私人共享文件 App。

第一期应严格聚焦于：

- 文件访问
- 文件传输
- 基础预览
- 简单权限显示

不建议在第一期引入：

- 传统 ECS
- 数据库
- 复杂后台
- 细粒度协作能力

该方案能够在保持架构简单和成本可控的前提下，满足当前核心需求，并为后续扩展保留空间。
